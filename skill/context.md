## 5. VIP-клуб: каналы и логика доступа

### 8 каналов внутри клуба

| Канал (channel) | Название | Режим |
|----------------|---------|-------|
| `intraday` | Интрадей / Сигналы | readonly (только автор пишет) |
| `chat` | Общий чат | full (все пишут) |
| `metals` | Металлы | full |
| `oil` | Нефть и газ | full |
| `products` | Продовольствие | full |
| `video` | Видео-разборы | readonly |
| `tech` | Тех. анализ | full |
| `knowledge` | База знаний | readonly |
| `access_info` | Информация о доступе | readonly |

### Роли пользователей
- `owner` — полный доступ, всегда активен
- `admin` — полный доступ, без подписки
- `editor` — редактор публичного контента
- `subscriber` — обычный пользователь, нужна активная подписка

### Логика доступа (hasAccess)
```
owner || admin → доступ всегда
subscriber → только если subscription.status === 'active' && expires_at > now
```

### Статусы подписки
- `pending` — заявка подана, чек загружен, ждёт проверки → страница NoAccess
- `active` — оплата подтверждена → полный доступ
- `expired` / `cancelled` → нет доступа

## 8. Авторизация и сессии

### Схема
1. Вход/регистрация → backend/auth → генерируется токен → сохраняется в club_sessions
2. Токен → в localStorage (ключ: `club_token`)
3. Каждый запрос → заголовок `X-Auth-Token: {token}`
4. Backend читает `event.get("headers", {}).get("X-Auth-Token", "")` — напрямую, без прокси
5. SQL: `SELECT ... WHERE s.token = %s AND s.expires_at > NOW()`

### Хеширование паролей
```python
salt = os.environ.get("PASSWORD_SALT", "tradeclub_salt_default")
password_hash = hashlib.sha256((password + salt).encode()).hexdigest()
```
**При миграции:** задать `PASSWORD_SALT=tradeclub_salt_default` в .env на Beget — все пароли продолжат работать.

## 11. Важные технические решения (сохранить при переносе)

### Заголовок авторизации
Используется **`X-Auth-Token`** (не `Authorization`). Это намеренное решение — Poehali.dev перехватывает `Authorization`, поэтому выбран кастомный заголовок. На Beget работает без изменений.

### Схема БД
Все таблицы в отдельной схеме `t_p67093308_rtrader_hub`. При переносе рекомендуется переименовать в `public` (стандарт PostgreSQL) и обновить `MAIN_DB_SCHEMA` в env.

### Simple Query Protocol
Все SQL-запросы используют psycopg2 в режиме Simple Query (без параметров `%s` в некоторых местах — значения вставляются напрямую). Не использовать asyncpg или Extended Query Protocol.

### S3 и CDN
- Endpoint для записи: `https://bucket.poehali.dev`
- CDN для чтения: `https://cdn.poehali.dev/projects/{AWS_ACCESS_KEY_ID}/bucket/`
- При переносе оба endpoint меняются на новый S3-сервис
- В БД хранятся полные CDN URL — после переноса нужен UPDATE по всем таблицам

### func2url.json
Frontend получает URL backend-функций из `backend/func2url.json`. При переносе обновить все 20 URL в этом файле — фронт автоматически подхватит новые адреса.

### Жёсткие CDN-ссылки в коде
В `index.html` и `public/manifest.json` захардкожены ссылки с Project UUID `39a4ac3c-06be-41f0-97f1-cc3959260964`. После переноса медиа на новый S3 — обновить в этих двух файлах.

### Скрипты Poehali.dev
В `index.html` подключены скрипты `cdn.poehali.dev/intertnal/js/pp-min-2.js`, `route-min.js`, `telemetry-min.js`, `inspector-min.js`. При переносе на Beget — **удалить** эти четыре строки.
